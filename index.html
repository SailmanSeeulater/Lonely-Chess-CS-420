Lonely Chess

A chess-based esoteric programming language where a valid PGN chess game becomes source code.

The website below includes a real browser version of the Lonely Chess runtime. Paste a PGN program, run it, and the interpreter will parse the moves using the same core rules as the Python runtime.

PGN is code

A PGN chess game file is the source program.

Board is memory

Variables are stored in interpreter state as slots like p_h2 and p_g2.

Pieces are syntax

Knights, rooks, bishops, kings, queens, and pawns all have programming roles.

Python + browser

The project has Python runtime files, and this page ports that logic into JavaScript for the browser.

Syntax Guide

How the language works

Lonely Chess is unusual, but the rules are consistent. Every chess move either changes the interpreter mode, writes data, performs math, controls flow, or prints output.

Integer mode

White Knight to a3 begins integer declaration.

String mode

White Knight to c3 begins string declaration.

Variable name

A pawn move selects a variable based on file, such as h-file becoming p_h2.

Binary encoding

Black Rook on rank 6 writes 7-bit values using b6 through h6.

Loop

White King to e2 starts or advances the loop; King to e1 prints the current iteration result.

If + modulo

Bishop opens the if block, rook opens modulo, black rook bounces count the divisor, and rook returns to a1 to evaluate.

{ === LONELY CHESS: int p_h2 = 100 then print(p_h2) === }

{ --- SETUP PHASE --- }
{ Move 1: W Knight b1->a3  signals: begin INTEGER instantiation mode }
{ Move 1: B pawn a7->a5    filler move }
1. Na3 a5

{ Move 2: W pawn h2->h3    declares variable name p_h2 (based on pawn home square h2) }
{ Move 2: B pawn a5->a4    filler move }
2. h3 a4

{ Move 3: W pawn h3->h4    second pawn push, no-op (var already declared) }
{ Move 3: B rook a8->a6    arms encoding mode (rook reaches 6th rank) }
3. h4 Ra6

{ --- ENCODING PHASE: writing 1100100 = 100 in 7-bit binary --- }
{ b6=bit6, c6=bit5, d6=bit4, e6=bit3, f6=bit2, g6=bit1, h6=bit0 }

{ Move 4: W Knight a3->c4  no-op wait }
{ Move 4: B rook a6->b6    sets bit6 -> acc = 1000000 }
4. Nc4 Rb6

{ Move 5: W Knight c4->a3  no-op wait }
{ Move 5: B rook b6->c6    sets bit5 -> acc = 1100000 }
5. Na3 Rc6

{ Move 6: W Knight a3->c4  no-op wait }
{ Move 6: B rook c6->f6    sets bit2 -> acc = 1100100 = 100 }
6. Nc4 Rf6

{ Move 7: W Knight c4->a3  no-op wait }
{ Move 7: B rook f6->a6    finalises value: 1100100 = 100 }
7. Na3 Ra6

{ --- FINALISE PHASE --- }
{ Move 8: W Knight a3->b1  commits p_h2 = 100 }
{ Move 8: B rook a6->a8    encoding complete, resets to start position }
8. Nb1 Ra8

{ --- PRINT PHASE: print(p_h2) --- }
{ Move 9:  W pawn h4->h5   selects p_h2 for printing (pawn home = h2) }
{ Move 9:  B pawn e7->e6   filler }
9. h5 e6

{ Move 10: W Queen d1->d2  initiates print (path now clear after pawn moved) }
{ Move 10: B pawn e6->e5   filler }
10. Qd2 e5

{ Move 11: W Queen d2->d1  finalises print -> OUTPUT: p_h2 = 100 }
{ Move 11: B pawn e5->e4   filler }
11. Qd1 e4

Runtime vs Interpreter

The runtime prints only final program output. The interpreter prints move-by-move annotations, which is better for debugging and explaining how each chess move changes the program state.
